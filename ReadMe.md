Пет-проект: парсер вакансий на разных сервисах
стэк: голанг (пока консольная реализация)

что сделано:

- создан каркас проекта
- реализован механизм поиска вакансий на 2х источниках (HH.ru, SuperJob.ru) (parser manager, можно добавлять новые источники поиска вакансий)
- реализован sharded inmemory cache - для поиска
- реализован sharded inmemory cache - для получения конкретной ваканси по ID из первого кэша (используем обратный индекс)
- реализован rate limiter для каждого парсера (ограничение частоты запросов на ресурс)
- сделан адекватный поиск по ID среди результатов
- реализован паттерн "семафор" для каждого парсера (для ограничения параллельности)
- реализован circuit breaker для каждого парсера (для блокировки недоступного сервиса)
- реализован механизм создания парсеров через паттерн "фабрика"
- отрефакторина логика получения конфиг-данных (из .env и из .yml)

перспектива:

- реализовать валидаию данных из источников config
- реализовать механизм восстановления данных при запросе ID, если в основном кэше данных нет ("lazy-валидация с фолбэком")
- реализовать глобальный семафор - ограничение одновременных поисков [в parse manager]
- реализовать приоритетные очереди - QoS для разных типов пользователей (платный/админ, обычный, боты) [в parse manager]
- реализовать Load balancer - распределение между источниками [в parse manager]
- реализовать глобальный Circuit Breaker - мониторинг общего здоровья [в parse manager]
- реализовать Resource manager - контроль памяти/CPU [в parse manager]
- реализовать очередь вызовов парсеров - ограничение параллельных вызовов [в parse manager]
- отредакторить код (разделить полностью на слои: DTL, service, repository)
- конкурентный парсинг из произольного количества источников
- пагинация результатов поиска
- кэширование результатов поиска в Redis (с TTL)
- логирование
- тесты

Планируемая архитектура:
Пользовательский запрос
↓
┌─────────────────────────────────┐
│ PARSER MANAGER УРОВЕНЬ ← Глобальный контроль
│ • Глобальный семафор
